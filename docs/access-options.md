# Buding access options

本文只评估接入方式，不改变当前代码逻辑。

## 手机接入

### 推荐方案：HTTPS Web 访问

手机和电脑在同一局域网时，可以让 Web 服务监听 `0.0.0.0`，手机访问：

```text
https://<电脑局域网 IP>:<端口>/
```

注意事项：

- 仅用 `http://192.168.x.x:<端口>` 可能能打开页面，但手机浏览器通常不会给麦克风权限。语音输入需要 HTTPS 安全上下文。
- 不建议直接把服务页暴露到公网。服务启动/停止接口默认限制 localhost，远程开放前应先加登录鉴权。
- 如果只想手机聊天，不想手机控制服务，可以保留服务控制接口的本地限制。

可选落地方式：

- 局域网 HTTPS 反向代理：用 Caddy/Nginx 在本机或服务器上代理到当前 FastAPI 服务。
- Tailscale/内网穿透：适合自己手机访问，外部攻击面更小。
- Cloudflare Tunnel/ngrok：适合临时外网测试，但正式使用仍建议加鉴权。
- PWA：在现有 Web 页面基础上加 manifest、图标、离线壳和移动端布局优化，让手机像 App 一样打开。

## 微信接入

### 个人微信号自动化

不推荐。个人号自动化通常稳定性和合规风险都较高，也容易被风控。除非只是个人本机实验，否则不建议作为正式接入方案。

### 微信公众号

可以接入，但更适合“文本问答/语音留言式问答”，不适合实时语音对话。

基本链路：

```text
用户消息 -> 微信服务器回调 -> Buding 后端 -> ASR/LLM/TTS -> 回复文本或媒体
```

限制与成本：

- 需要一个可公网 HTTPS 访问的回调地址。
- 文本消息接入较直接；语音消息需要先从微信媒体接口拉取音频，再送 ASR。
- 被动回复有时效限制，复杂长回答需要客服消息、模板/订阅消息或异步方案。
- TTS 音频回复需要处理媒体上传、格式、时长和缓存。

### 企业微信 / 微信客服

适合内部使用或客服场景，整体比个人号自动化更合规。可以做文本消息、会话回调、客服坐席式转接。语音实时对话仍然不是最自然的形态。

### 微信小程序

如果目标是手机端语音体验，这是微信生态内最推荐的路线。

基本链路：

```text
小程序录音 -> HTTPS/WebSocket -> Buding 后端 -> ASR/LLM/TTS -> 小程序播放音频
```

优势：

- 官方支持录音、播放、WebSocket 和 HTTPS 请求。
- 可以做接近当前网页的实时语音体验。
- 后续能加登录、用户记忆隔离、会话列表、通知等。

成本：

- 需要小程序账号、域名备案/HTTPS、接口域名配置。
- 前端需要单独写小程序页面，不能直接复用全部浏览器 Web API。
- 音频格式、分片上传、播放缓冲要单独适配。

## 建议路线

1. 先做手机 HTTPS Web/PWA：最快验证移动端体验。
2. 再做用户鉴权和远程访问安全：避免服务控制接口裸露。
3. 如果确认要进微信生态，优先评估小程序。
4. 公众号/企业微信作为文本入口或客服入口，不作为实时语音主入口。

## 需要进一步确认

- 手机是否只给自己使用，还是多人使用？
- 是否必须在公网访问？
- 微信接入是要实时语音、文本聊天，还是公众号客服式消息？
- 是否已有备案域名、HTTPS 证书、小程序或公众号主体？
