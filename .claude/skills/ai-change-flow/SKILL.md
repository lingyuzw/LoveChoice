# AI Change Flow

你是主控 AI。用户给出任何开发需求、前端美化需求、功能添加需求、重构需求、Bug 修复需求时，必须按以下流程工作。

## 铁律（违反任何一条 = 流程失败）

1. **每个阶段必须输出一段最终文字给用户看。** 不能只在内部思考，必须把该阶段的结论用文字呈现出来。
2. **每个阶段输出后必须等待用户确认。** 用户说"可以"/"继续"/"没问题"之后才能进入下一阶段。不得自动跳过。
3. **禁止跳过阶段。** 不管任务多简单、多明显，0→1→2→3→4→5 一个不能少。
4. **阶段 1-2 禁止改代码。** 只读、只分析。阶段 3 才开始动手。

## 工作流总览

```
用户需求
→ 阶段 0: Skill Discovery（检查/搜索缺失 skills）
→ 用户确认是否安装
→ 阶段 1: 需求确认
→ 用户确认
→ 阶段 2: 方案设计
→ 用户确认
→ 阶段 3: 执行修改
→ 阶段 4: 代码审查
→ 必要返工
→ 阶段 5: 最终文档
```

## 启动流

每次被触发时，**第一句话必须输出**（不要放在 thinking 块里）：

```
## AI Change Flow 启动

我会严格按以下流程处理，每阶段完成后等你确认：

0. Skill Discovery — 检查需要的 skills
1. 需求确认 — 复述需求、分析影响、确认目标
2. 方案设计 — 给方案、文件级计划、风险点
3. 执行修改 — 按方案改代码
4. 代码审查 — 检查质量、防回归
5. 最终文档 — 修改说明、测试方法

现在进入阶段 0。
```

然后立即进入阶段 0。

---

## 阶段 0：Skill Discovery

**目的：** 检查当前任务需要哪些 skills，缺的去搜索。

**输出格式**（必须输出这段文字给用户）：

```
## 阶段 0：Skill Discovery

### 当前任务类型
...

### 本地已有 skills
...

### 建议使用的 skills
...

### 缺失的 skills
...

### 搜索建议
...

是否安装缺失的 skills？（确认后进入阶段 1）
```

**规则：**
- 检查 `.claude/skills/` 和全局 skills 目录。
- 如果缺 skill，用 WebSearch 搜索可信来源（github.com/anthropics/skills 等）。
- 不自动安装，只给建议。
- 用户确认后进入阶段 1。用户说"跳过"也可以直接进阶段 1。

---

## 阶段 1：需求确认

**目的：** 分析需求、明确目标、确认范围。

**输出格式**（必须输出这段文字给用户）：

```
## 阶段 1：需求确认

### 我理解的需求
（用 3-5 句话复述用户真正想要什么）

### 模糊点
（如果有不明确的地方，列出来；没有就说"无模糊点"）

### 必须实现
...

### 可选优化
...

### 可能影响范围
（涉及哪些文件、模块、接口）

### 确认问题
（有问题就问；没有就说"无需额外确认"）

### 下一步
以上理解是否正确？确认后进入阶段 2（方案设计）。
```

**规则：**
- 可以读代码，不能改代码。
- 如果需求模糊，必须提出问题等用户澄清。
- 用户确认后进入阶段 2。

---

## 阶段 2：方案设计

**目的：** 给出清晰可执行的修改方案。

**输出格式**（必须输出这段文字给用户）：

```
## 阶段 2：方案设计

### 总体思路
（一段话说明怎么实现、为什么这样做）

### 本次必须完成
...

### 建议顺手优化
（区分哪些是"本次做"哪些是"以后做"）

### 文件级修改计划
| 文件 | 修改内容 | 风险 |
|------|---------|------|
| ... | ... | ... |

### 不应修改的内容
（明确列出哪些 API、DOM ID、配置字段、WebSocket 协议不能动）

### 测试计划
...

### 下一步
方案是否可行？确认后进入阶段 3（执行修改）。
```

**规则：**
- 不能改代码，只设计方案。
- 区分"必须做"和"建议做"。
- 用户确认后进入阶段 3。

---

## 阶段 3：执行修改

**目的：** 按确认方案修改代码。

**输出格式**（修改完成后输出这段文字给用户）：

```
## 阶段 3：执行修改

### 修改文件清单
| 文件 | 修改说明 |
|------|---------|
| ... | ... |

### 是否影响原有接口
...

### 快速验证方法
...

修改完成。现在进入阶段 4（代码审查）。
```

**规则：**
- 修改前先读相关文件。
- 不擅自扩大范围。
- 不改后端 API、WebSocket、配置字段名。
- 复杂逻辑加注释。
- 修改完自动进入阶段 4（审查不需要用户确认，但审查结果要输出）。

---

## 阶段 4：代码审查

**目的：** 独立审查修改质量。

**输出格式**（必须输出这段文字给用户）：

```
## 阶段 4：代码审查

### 结论
（通过 / 需要返工）

### 需求完成度
（是否覆盖了阶段 1 的全部必须项）

### 是否破坏现有功能
（API、WebSocket、配置、DOM ID 引用等）

### 代码质量问题
（有就列；没有就说"无"）

### 注释检查
（有就列；没有就说"无"）

### 返工项
（有就列，没有就说"无返工项"）

### 测试建议
...
```

**规则：**
- 只审查，不改代码（除非发现严重问题且用户同意）。
- 如果发现严重问题，说明需要返工的内容，等用户指示。
- 审查通过后进入阶段 5。

---

## 阶段 5：最终文档

**目的：** 输出完整修改文档，方便用户了解全貌。

**输出格式**（必须输出这段文字给用户）：

```
## 阶段 5：最终文档

# 修改完成说明

### 本次目标
...

### 修改文件清单
| 文件 | 修改说明 |
|------|---------|

### 新增功能
...

### UI / 交互优化
...

### 是否影响原有接口
...

### 启动和测试方法
...

### 已知风险
...

### 后续优化建议
...

### 建议提交信息
git commit -m "..."
```

## 强制规则总结

1. 每个阶段必须输出最终文字，不能只在 thinking 块里推理。
2. 每个阶段输出后必须等待用户确认（阶段 4 除外，审查完成自动进阶段 5）。
3. 五个阶段一个不能跳，任务再简单也得走完。
4. 阶段 0-2 禁止改代码，只读只分析。
5. 阶段 3 严格按方案执行，不扩大范围。
6. 阶段 4 独立审查，不在执行时偷跑。
7. 阶段 5 输出完整文档。
8. 所有输出用中文。
9. 不确定就说"不确定"，不猜。
10. 优先保持项目简单清晰，不过度工程化。
